08. Colaboración en equipo

Objetivo

Aprender a compartir configuración, comandos personalizados y agentes con otros miembros del equipo.

Duración estimada: 25 minutos

Consejo¿Por qué compartir configuración?

Si trabajas en equipo, cada miembro puede tener sus propias preferencias. Pero la configuración del proyecto (agentes, comandos, scripts) debe ser la misma para todos. Así todos trabajan con las mismas herramientas.

Compartir configuración

OpenCode permite exportar e importar la configuración del proyecto, de modo que todo el equipo pueda usar los mismos agentes, comandos y skills.

La clave está en versionar la configuración junto con el proyecto. Todo lo que está dentro del directorio del proyecto se puede compartir con Git: agentes, opencode.jsonc, mcp.json y scripts.

Lo que no se comparte es la configuración global (~/.config/opencode/), como skills personales o claves de API. Para más detalles sobre proveedores y seguridad de claves, consulta el paso 03.

Analogía: piensa en la configuración del proyecto como una receta de cocina. Si todos los cocineros usan la misma receta, el plato saldrá igual. Si cada uno improvisa, el resultado será impredecible.

Qué se comparte y qué no

Elemento Se comparte Dónde está
Agentes personalizados ✅ .opencode/agents/
opencode.jsonc del proyecto ✅ Raíz del proyecto
mcp.json ✅ .opencode/mcp.json
Scripts de automatización ✅ *.sh en la raíz
Skills reutilizables ❌ ~/.config/opencode/skills/
Claves de API ❌ No deben versionarse
Preferencias personales ❌ ~/.config/opencode/
NotaCompartir la configuración

Compartir la configuración con Git es el método recomendado, pero no el único. Puedes copiar el directorio del proyecto por red, USB o cualquier otro medio. Lo importante es que la configuración de OpenCode viaje siempre con el código.

Manos a la obra

Ejercicio 1 — Preparar el proyecto para compartir

Asegúrate de que todo lo compartible está dentro del repositorio:

git status

Deberías ver los agentes, opencode.jsonc y demás archivos de configuración (el mcp.json está dentro de .opencode/). Si falta alguno, añádelo:

git add .opencode/ opencode.jsonc
git commit -m "Añade configuración compartible del equipo"

Ejercicio 2 — Crear un .gitignore para claves

Añade estas líneas a .gitignore para evitar filtrar información sensible:

.env
*.key
**/secrets*

Si tienes un mcp.json que contiene rutas locales o claves, considera crear una plantilla mcp.template.json sin datos sensibles:

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "RUTA_DEL_EQUIPO"
      ]
    }
  }
}

Cada miembro copia la plantilla, la renombra a .opencode/mcp.json y ajusta la ruta a su máquina.

Ejercicio 3 — Documentar la configuración

Crea un archivo COMPARTIR.md (o añádelo al README.md) con las instrucciones para el equipo:

## Configuración del proyecto

1. Clona el repositorio
2. Copia `mcp.template.json` como `mcp.json`
3. Ajusta las rutas en `mcp.json` a tu máquina
4. Si usas skills, cópialos desde el directorio compartido

Comandos disponibles:
- `bash render-all.sh` — genera los HTML del curso
- `bash revisar-todo.sh` — revisa todos los archivos

Ejercicio 4 — Simular la incorporación de un nuevo miembro

Clona el repositorio en otro directorio para ver qué recibe un compañero:

cd /tmp
git clone /ruta/del/proyecto curso-clonado
cd curso-clonado

Abre OpenCode en ese directorio y comprueba que los agentes están disponibles:

@revisor-estilo ¿Qué puedes hacer?

Si los agentes están en .opencode/agents/, OpenCode los reconocerá automáticamente.

Ejercicio 5 — Compartir skills manualmente

Aunque los skills no se versionan con Git, puedes compartir el directorio del skill para que cada miembro la copie:

cp -r ~/.config/opencode/skills/formatear-fechas /tmp/compartir/

O mejor: crea un script instalar-skills.sh en el proyecto:

#!/bin/bash
# Copia los skills compartidos a la configuración local
SKILLS_DIR="$HOME/.config/opencode/skills"
mkdir -p "$SKILLS_DIR"
cp -r skills-compartidos/* "$SKILLS_DIR"
echo "Skills instalados. OpenCode los detectará automáticamente."

Y guarda los skills en skills-compartidos/ dentro del repositorio:

mkdir skills-compartidos
cp -r ~/.config/opencode/skills/formatear-fechas skills-compartidos/
git add skills-compartidos instalar-skills.sh
git commit -m "Añade skills compartidos y script de instalación"

Ejercicio 6 — Estandarizar comandos del equipo

Define comandos personalizados en opencode.jsonc para que todo el equipo tenga las mismas tareas rápidas:

{
  "commands": {
    "revisar-estilo": {
      "command": ["bash", "revisar-todo.sh"],
      "description": "Revisa el estilo de todos los documentos"
    },
    "generar-html": {
      "command": ["bash", "render-all.sh"],
      "description": "Genera todos los HTML del curso"
    },
    "status": {
      "command": ["git", "status"],
      "description": "Muestra el estado del repositorio"
    }
  }
}

Ahora cualquier miembro del equipo puede ejecutar:

opencode run "ejecuta el comando generar-html"

Flujo de trabajo en equipo

1. git pull          → Trae los cambios del equipo
2. Editas documentos  → Trabajas con OpenCode
3. git add + commit  → Guardas tus cambios
4. git push          → Compartes con el equipo

Cada miembro tiene su configuración local (skills, claves) y la configuración compartida (agentes, comandos, scripts) viene con el repositorio.

Buenas prácticas

  • No versiones las claves — usa plantillas y .gitignore

  • Documenta el onboarding — explica en el README qué hacer al unirse al proyecto

  • Usa comandos personalizados — evita que cada miembro tenga que recordar rutas y scripts

  • Revisa los permisos — si el equipo crece, revisa quién puede modificar la configuración compartida
