07. Permisos: controla acciones (permissions)

Objetivo

Aprender a controlar qué puede y qué no puede hacer OpenCode en entornos con documentos sensibles.

Duración estimada: 20 minutos

Consejo¿Por qué aprender sobre permisos?

Controlar qué puede y qué no puede hacer OpenCode te da tranquilidad. No es solo para empresas: si trabajas con documentos importantes, los permisos evitan errores costosos.

Si no usas Git para versionar tus documentos, esto es aún más crítico: sin historial de cambios, una modificación accidental puede ser irreversible. Usar tirantes y correa siempre aumenta la seguridad.

¿Por qué controlar permisos?

A veces es conveniente limitar lo que OpenCode puede hacer. No por desconfianza, sino por prudencia: evitar modificaciones accidentales, proteger archivos importantes o simplemente trabajar más tranquilo.

Ejemplos cotidianos:

  • Evitar borrados accidentales — si estás revisando un documento importante, puedes denegar la edición para que OpenCode solo lea y sugiera cambios, sin tocar el original

  • Proteger archivos privados — si tienes un directorio con datos personales, puedes impedir que OpenCode la lea

  • Limitar comandos del sistema — si no necesitas que OpenCode ejecute comandos bash, puedes desactivarlos

  • Trabajar en equipo sin riesgos — que cada miembro solo pueda modificar ciertos directorios

Documento protegido: cuando un documento está terminado y no debe modificarse, puedes cambiar su permiso a solo lectura. Así OpenCode puede consultarlo pero no alterarlo.

Analogía: piensa en los permisos como las llaves de una casa. No todas las puertas necesitan la misma llave. A OpenCode le das las llaves que necesita para trabajar, ni más ni menos.

Niveles de permisos

Nivel Dónde se define Ámbito
Global ~/.config/opencode/opencode.jsonc Todos los proyectos
Proyecto opencode.jsonc (o opencode.json) en la raíz Solo ese proyecto
Agente .opencode/agents/*.md Solo ese agente

Los permisos más restrictivos ganan. Si el proyecto deniega la edición, ningún agente puede editar archivos aunque lo pida. Si hay conflictos entre allow y deny para una misma acción, el deny siempre prevalece.

Acciones controlables

Acción Descripción
edit Modificar archivos existentes
write Crear archivos nuevos
bash Ejecutar comandos en la terminal
read Leer archivos (restringible)
browser Navegar por internet

Manos a la obra

Ejercicio 1 — Ver la configuración actual

Abre ~/.config/opencode/opencode.jsonc (o créalo si no existe) y mira su contenido:

cat ~/.config/opencode/opencode.jsonc

Si no existe, no pasa nada: OpenCode usa los valores por defecto (todo permitido).

Ejercicio 2 — Restringir la edición en el proyecto

Crea un archivo opencode.jsonc en la raíz del curso:

{
  "permission": {
    "edit": "deny"
  }
}

Guarda el archivo, cierra y vuelve a abrir OpenCode. Ahora, aunque le pidas a OpenCode que modifique un archivo, no podrá hacerlo.

Pruébalo en Build:

Añade una línea al final de index.qmd

OpenCode debería responder que no tiene permiso para editar.

Ejercicio 3 — Permitir solo lectura de ciertos directorios

A veces quieres que OpenCode lea archivos de un directorio pero no de otras:

{
  "permission": {
    "read": {
      "allow": ["docs/", "glosario/"],
      "deny": ["privado/", "confidencial/"]
    }
  }
}

Crea un directorio privado/ en la raíz del proyecto con un archivo de prueba y comprueba que OpenCode no puede leerlo.

Ejercicio 4 — Denegar bash pero permitir edición

Si tu equipo usa OpenCode para redactar pero no quieres que ejecute comandos en el sistema:

Modifica el opencode.jsonc que creaste en el Ejercicio 2 y sustitúyelo por:

{
  "permission": {
    "bash": "deny",
    "edit": "allow"
  }
}

Pruébalo: pide a OpenCode que liste los archivos con bash:

Lista los archivos de este directorio con bash

Debería responder que no tiene permiso.

Ejercicio 5 — Permisos por agente

Vuelve al agente revisor-estilo del paso 01. Su archivo .opencode/agents/revisor-estilo.md ya tiene permisos:

---
description: Revisa estilo, tono y claridad sin modificar nada
mode: subagent
permission:
  edit: deny
---

Este agente no puede editar aunque el proyecto lo permita. Los permisos del agente siempre se aplican.

Añade también bash: deny para que no pueda ejecutar comandos:

---
description: Revisa estilo, tono y claridad sin modificar nada
mode: subagent
permission:
  edit: deny
  bash: deny
---

Ejercicio 6 — Combinar niveles de permisos

Crea una jerarquía completa de ejemplo:

  • Global: bash permitido, edit permitido
  • Proyecto (curso): edit permitido, bash denegado
  • Agente revisor: edit denegado

Resultado: el agente revisor no puede editar (por su propio permiso) ni ejecutar bash (por el proyecto). El resto de agentes sí pueden editar, pero ninguno puede ejecutar bash.

Ejercicio 7 — Proteger un documento terminado

Imagina que paso-01.qmd ya está revisado y no debe modificarse. Puedes protegerlo creando un archivo opencode.jsonc en la raíz:

{
  "permission": {
    "read": {
      "deny": ["paso-01.qmd"]
    }
  }
}

Ahora OpenCode no podrá leer ese archivo específico. Prueba a pedirle que lo revise:

Revisa el estilo de paso-01.qmd

Debería indicar que no tiene permiso para leerlo.

Buenas prácticas

  • Principio de mínimo privilegio — otorga solo los permisos necesarios para cada tarea

  • Documenta los permisos — incluye comentarios en opencode.jsonc explicando por qué existen restricciones

  • Prueba cada restricción — verifica que el comportamiento es el esperado antes de desplegar

  • Revisa periódicamente — los permisos deben revisarse igual que cualquier otra configuración

ConsejoPide a OpenCode que configure los permisos

Ahora que entiendes cómo se controlan los permisos y para qué sirven, puedes pedirle a OpenCode que realice los ajustes pertinentes. Por ejemplo, escribe en Plan:

“Crea un archivo opencode.jsonc que deniegue bash pero permita edición, y que solo pueda leer el directorio docs/”

OpenCode generará la configuración por ti.
