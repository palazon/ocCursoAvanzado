04. Control de versiones (git)

Objetivo

Aprender los fundamentos de Git para versionar documentos, mantener un historial de cambios y colaborar sin riesgo.

Duración estimada: 20 minutos

NotaGit funciona en tu ordenador

Todo lo que verás en este paso es local: no necesitas conexión a internet ni cuenta en ningún servidor. Git guarda el historial en tu propio directorio.

¿Por qué Git si no soy programador?

Git permite guardar versiones de tus documentos, ver qué cambió entre una versión y otra, y deshacer cambios sin perder trabajo. Es la herramienta estándar para colaborar en proyectos de documentación.

Si alguna vez has tenido archivos como informe-v3-final-definitivo.docx, Git es la solución. Con Git:

  • Cada cambio queda registrado con fecha, autor y motivo
  • Puedes volver a cualquier versión anterior
  • Podrías colaborar contigo mismo o con otros sin pisar los archivos
  • Sabes exactamente qué cambió entre versiones

Piensa en Git como un álbum de fotos: cada commit es una foto de tus archivos en un momento dado. Si algo sale mal, puedes volver atrás.

Git funciona en tu ordenador. No necesitas internet ni cuentas externas. Solo tú controlas tus documentos.

Conceptos clave

Concepto Qué significa
Repositorio El directorio donde Git guarda el historial
Commit Una foto de tus archivos en un momento dado
Historial La lista de commits ordenados en el tiempo

No necesitas memorizar todo. Con cuatro comandos haces el 90 % del trabajo, y en casi todo puede ayudarte OpenCode.

Manos a la obra

Vamos a crear un proyecto desde cero con buenas prácticas.

Ejercicio 1 — Configurar Git por primera vez

Abre una terminal y escribe tu nombre y correo. Git los usará para firmar tus commits:

git config --global user.name "Tu Nombre"
git config --global user.email "tu@correo.com"
NotaEsto no es necesario para trabajar en local

Si solo trabajas en tu ordenador, no es estrictamente necesario. Pero si algún día colaboras con otros (o incluso contigo mismo desde otro equipo), Git usará estos datos para identificar quién ha hecho cada cambio.

Tu mejor amigo del futuro eres tú. Y Git puede ayudarte a serlo aún más.

Ejercicio 2 — Crear directorio y repositorio

mkdir ~/mi-proyecto
cd ~/mi-proyecto
git init

Esto crea un directorio vacío con un repositorio Git. El directorio oculto .git/ guardará todo el historial. No lo toques manualmente.

Ejercicio 3 — Crear .gitignore antes del primer commit

Buena práctica: crea un archivo .gitignore para que Git ignore archivos que no necesitas versionar. Así reduces el número de ficheros que Git guarda en cada nueva versión:

*.html
.DS_Store
*.bak
git add .gitignore
git commit -m "Añade .gitignore para ignorar archivos temporales"
Consejo¿Por qué ignorar archivos?

Los archivos temporales, HTML generados u otros que se recrean automáticamente no necesitan guardarse en el historial. Así Git trabaja más rápido y el historial es más limpio.

Ejercicio 4 — Primer commit

Como ejemplo, creamos un fichero desde la terminal. En el día a día, OpenCode irá guardando los nuevos ficheros y los cambios que se produzcan en una sesión de trabajo:

echo "# Mi documento" > documento.md
git add documento.md
git commit -m "Primer commit: documento inicial"

El comando git add prepara el archivo. El comando git commit lo guarda en el historial con un mensaje descriptivo.

ConsejoLa importancia del mensaje en el commit

Un buen mensaje te ayuda a entender qué hiciste y por qué, aunque hayan pasado meses. Evita mensajes vagos como “cambio” o “update”. Mejor: “Añade sección de conclusiones” o “Corrige erratas en capítulo 2”.

Ejercicio 5 — Ver el historial

En ocasiones quieres ver lo que has ido guardando, de forma global:

git log --oneline

Deberías ver algo como:

a1b2c3d Primer commit: documento inicial
b4e5f6g Añade .gitignore para ignorar archivos temporales

Ejercicio 6 — Hacer un cambio y ver la diferencia

Como ejemplo, añadimos una línea al fichero existente en el directorio del repositorio. Para verlo más fácil, usamos un comando:

echo "Segunda línea del documento" >> documento.md
git diff

Git te muestra exactamente qué líneas has añadido (verde) y cuáles has quitado (rojo).

Ejercicio 7 — Segundo commit

Cuando estés satisfecho con el cambio:

git add documento.md
git commit -m "Añade segunda línea al documento"

Ejercicio 8 — Ver el estado

Antes de cada commit, revisa qué hay pendiente:

git status

Te dice qué archivos están modificados, qué está en el staging (una zona de preparación donde seleccionas qué quieres incluir en el próximo commit) y qué no está registrado.

Consejo¿Y si me equivoco?

Si has modificado un archivo y no te gusta el resultado, puedes volver al estado del último commit:

git restore documento.md

Este comando descarta los cambios no guardados.

Flujo de trabajo diario para documentalistas

1. git status         → ¿Qué ha cambiado?       (opcional)
2. git diff           → ¿Qué exactamente?        (opcional)
3. git add archivo    → Preparo el cambio         (necesario)
4. git commit -m "..." → Lo guardo en el historial (necesario)

Los dos primeros pasos son opcionales pero muy útiles: te ayudan a revisar qué vas a guardar. Los dos últimos son siempre necesarios.

Comandos útiles

Comando Para qué sirve
git status Ver el estado actual del repositorio
git diff Ver cambios sin commited
git log --oneline Historial resumido
git restore archivo Deshacer cambios en un archivo
git show HASH Ver qué cambió en un commit concreto

Resumen del paso

  • Has configurado Git en tu ordenador (local, sin servidor)
  • Has creado un repositorio con buenas prácticas (.gitignore primero)
  • Has visto cómo cambian los archivos con git diff
  • Tienes un flujo de trabajo diario para documentación
  • OpenCode puede ayudarte con los comandos de Git
